1. Introducción y Alcance del Proyecto

RocaNET es un portal web que permite a los clientes de la empresa consultar en tiempo real el stock disponible de productos cerámicos, visualizar imágenes, descargar fichas técnicas, y realizar solicitudes de material.

El sistema se alimenta de los datos del ERP existente de la empresa, sincronizando periódicamente las tablas de artículos, stocks, clientes, pedidos, albaranes y facturas hacia una base de datos replicada en un servidor dedicado. Esta arquitectura de base de datos espejo garantiza el aislamiento total del ERP de producción y permite que el portal web funcione de forma independiente.

Funcionalidades principales

  • Consulta de stocks en tiempo real con filtros avanzados (formato, serie, calidad, tono, calibre)
  • Visualización de imágenes de productos
  • Descarga de fichas técnicas en PDF
  • Sistema de propuestas / solicitudes de material con generación de PDF y envío por email
  • Consulta de pedidos, albaranes y facturas
  • Dashboard de estadísticas para administradores
  • Gestión de usuarios con roles (usuario, administrador, superusuario)
  • Soporte multi-idioma (español, inglés, francés)
  • Personalización visual por empresa (logo, favicon, tema de colores)
  • Acceso vía API REST para integraciones externas

2. Arquitectura del Sistema

El sistema utiliza una arquitectura de base de datos espejo (replicada). Se despliega un SQL Server en el servidor Linux (Docker) y se replican periódicamente las tablas del ERP. La aplicación web lee exclusivamente de la copia local. Esta arquitectura ofrece las siguientes ventajas:

  • Aislamiento total del ERP: La aplicación web nunca accede directamente al servidor de producción, eliminando cualquier riesgo de impacto en el ERP
  • Alta disponibilidad: El portal web funciona aunque el servidor ERP esté apagado o en mantenimiento
  • Mejor rendimiento: Las consultas web se ejecutan sobre una base de datos optimizada para lectura, sin competir con las operaciones del ERP
  • Seguridad: No es necesario exponer el servidor ERP a Internet ni abrir puertos adicionales en él

4. Datos Necesarios del ERP

El sistema necesita acceso de solo lectura a las siguientes áreas del ERP:

4.1 Datos maestros

  • Empresas: Código y nombre de las empresas gestionadas
  • Artículos: Catálogo completo de productos (código, descripción, formato, modelo, unidad, estado web)
  • Formatos: Dimensiones y tipos de producto
  • Calidades: Niveles de calidad
  • Modelos: Series/colecciones de producto
  • Colores: Catálogo de colores
  • Tonos y Calibres: Variaciones de tono y calibre
  • Unidades de medida: M2, ML, unidades, etc.
  • Pallets: Tipos de pallet y configuraciones
  • Clientes: Datos de clientes (razón social, CIF, email, dirección)
  • Países y Provincias: Tablas geográficas

4.2 Datos de stock y almacén

  • Stock por ubicación: Existencias por artículo, calidad, tono, calibre, pallet
  • Configuración de empaquetado: Piezas por caja, cajas por pallet, pesos
  • Códigos EAN13: Códigos de barras por artículo y calidad
  • Ubicaciones de almacén: Mapa de ubicaciones

4.3 Datos comerciales

  • Pedidos: Cabecera y líneas de pedidos de venta
  • Albaranes: Cabecera y líneas de albaranes de entrega
  • Facturas: Cabecera y líneas de facturas

4.4 Imágenes y documentos

  • Imágenes de artículos: Fotografías de producto (JPG). Pueden estar en tabla de base de datos o en carpeta de red
  • Fichas técnicas: Documentos PDF con especificaciones técnicas (opcional)

6. Imágenes de Artículos

Las imágenes de los artículos pueden provenir de dos fuentes según la instalación del ERP:

6.1 Opción A: Tabla en base de datos

Las imágenes se almacenan como campo binario (IMAGE/VARBINARY) en una tabla del ERP. Se sincronizan junto con el resto de tablas mediante BCP.

6.2 Opción B: Carpeta de red

Las imágenes se almacenan como ficheros JPG en una unidad de red compartida. El formato del nombre de fichero es:

CODIGO_ARTICULO@NUMERO.jpg Ejemplos: 0000000000004@1.jpg (primera imagen del artículo 0000000000004) 0000000000004@2.jpg (segunda imagen del mismo artículo) 0000000000012@1.jpg (primera imagen del artículo 0000000000012)

El script de sincronización:

  • Lee solo la imagen principal (@1) de cada artículo marcado como visible en web
  • Redimensiona las imágenes a un máximo de 800px de ancho para optimizar el almacenamiento
  • Solo inserta imágenes nuevas (no borra ni reemplaza las existentes)
  • Utiliza PowerShell para la lectura de ficheros y la inserción en base de datos
Requisito: El equipo que ejecuta la sincronización debe tener acceso a la unidad de red donde se almacenan las imágenes (por ejemplo Z:\).

8. Proceso de Instalación

8.1 Preparación del servidor Linux

  1. Instalar Ubuntu Server 22.04/24.04 LTS
  2. Configurar IP fija, DNS y zona horaria (Europe/Madrid)
  3. Instalar Docker y Docker Compose
  4. Crear usuario de aplicación
  5. Abrir puertos necesarios en firewall (22, 80, 443, 1433)

8.2 Despliegue de SQL Server (Docker)

docker run -d \ --name sqlserver2025 \ --restart unless-stopped \ --network jobers-net \ -e 'ACCEPT_EULA=Y' \ -e 'MSSQL_SA_PASSWORD=<contraseña_segura>' \ -e 'TZ=Europe/Madrid' \ -p 1433:1433 \ -v sqldata:/var/opt/mssql \ mcr.microsoft.com/mssql/server:2025-latest
Zona horaria: Es imprescindible configurar TZ=Europe/Madrid tanto en el contenedor de SQL Server como en el de la aplicación. Sin esta configuración, las fechas se almacenarán en UTC.

8.3 Creación de bases de datos

  1. Base de datos ERP replicada (ej: RocaNET_ERP): Contendrá las tablas sincronizadas del ERP
  2. Base de datos de aplicación (ej: ApiRestStocks): Contendrá usuarios, parámetros, configuración, y las vistas SQL
  3. Base de datos central (ej: jobers_control_usuarios): Gestión centralizada de conexiones multi-empresa

8.4 Despliegue de la aplicación (Docker Compose)

La aplicación se despliega con Docker Compose incluyendo:

  • Contenedor backend (Python + Gunicorn)
  • Contenedor Redis (caché y sesiones)
  • Volumen para el frontend (servido por el backend)

8.5 Configuración de Nginx (proxy inverso + SSL)

  • Redirección HTTP a HTTPS
  • Certificado SSL vía Let's Encrypt (certbot)
  • Proxy pass al backend en puerto 5000

9. Sincronización de Datos

La sincronización se ejecuta desde un equipo dedicado en la red local que se encarga de transferir automáticamente los datos del ERP al servidor de producción mediante un proceso programado.

9.1 Proceso de sincronización

  1. Verificación de conexiones: Comprueba conectividad con servidor origen y destino
  2. Comparación de esquemas: Si el número de columnas de una tabla ha cambiado en el origen, recrea la tabla en el destino automáticamente (DROP + CREATE)
  3. Comparación de datos: Calcula CHECKSUM_AGG de cada tabla en origen y destino. Si difiere, ejecuta TRUNCATE + BCP para actualizar
  4. Imágenes: Compara ficheros de la carpeta de red con registros existentes. Solo inserta las imágenes nuevas
  5. Finalización: Compacta el log de transacciones y registra la fecha de última sincronización

9.2 Frecuencia recomendada

FrecuenciaUso recomendado
Cada 30 minutosEntornos con actualización frecuente de stock
Cada 2-4 horasEntornos con volumen moderado
1 vez al día (noche)Entornos con poco cambio o tablas muy grandes
Nota: Las tablas se leen con WITH (NOLOCK) para minimizar el impacto en el rendimiento del ERP origen. No obstante, la sincronización de tablas muy grandes (>50.000 registros) puede generar carga momentánea en el servidor ERP.

9.3 Compatibilidad BCP

El script utiliza BCP en formato -w (wide character) que garantiza compatibilidad entre diferentes versiones de SQL Server (2008, 2012, 2019, 2025) y entre diferentes plataformas.

10. Seguridad

La seguridad es un pilar fundamental de la arquitectura de RocaNET. Se han implementado múltiples capas de protección que cubren desde la comunicación de red hasta la autenticación de usuarios y la protección de datos.

10.1 Seguridad en la comunicación

MedidaDescripción
HTTPS obligatorioTodo el tráfico web se cifra con TLS mediante certificado Let's Encrypt. Nginx redirige automáticamente HTTP a HTTPS
Conexión cifrada a BDLas conexiones a SQL Server usan Encrypt=yes con TLS, tanto desde la aplicación como desde la sincronización
Puertos internos no expuestosEl backend (5000) y Redis (6379) solo son accesibles dentro de la red Docker. Solo Nginx (80/443) está expuesto a Internet
SQL Server 1433 solo red localEl puerto de base de datos solo acepta conexiones desde la red interna, nunca desde Internet

10.2 Autenticación y gestión de sesiones

MedidaDescripción

Cookies segurasHttpOnly (protección XSS), Secure (solo HTTPS en producción), SameSite=Lax (protección CSRF)
Sesiones con trackingCada sesión se registra en base de datos con IP, user-agent y fecha. Los administradores pueden ver y cerrar sesiones activas

Cambio de contraseña obligatorioLos usuarios creados por un administrador deben cambiar su contraseña en el primer inicio de sesión
Verificación de emailLos usuarios registrados deben verificar su email mediante un token con expiración de 24 horas antes de poder acceder

10.7 Seguridad de la arquitectura espejo

Aislamiento del ERP: La arquitectura de base de datos espejo aporta una capa adicional de seguridad crítica. El servidor ERP de producción nunca está expuesto a Internet. La aplicación web solo accede a la copia replicada en el servidor Linux. Aunque un atacante comprometiera el portal web, no tendría acceso directo al ERP ni podría modificar datos de producción, ya que la sincronización es unidireccional (solo lectura desde el ERP).

10.8 Configuración segura por variables de entorno

Todos los datos sensibles se gestionan mediante variables de entorno, nunca hardcodeados en el código:

VariableFunción
SECRET_KEYClave secreta para firmar sesiones
CORS_ORIGINSOrígenes permitidos para peticiones CORS
DB_SERVER, DB_NAME, DB_USER, DB_PASSWORDCredenciales de conexión a base de datos
REDIS_URLURL de conexión a Redis

Buena práctica: En producción, las variables se definen directamente en la sección environment: del docker-compose.yml, no en ficheros .env, para garantizar que los workers de Gunicorn las cargan correctamente.

11.2 NAT / Port Forwarding (router)

Para acceso desde Internet, el router de la empresa debe redirigir los siguientes puertos a la IP interna del servidor Linux:

  • Puerto 22 (SSH) - opcional, solo para administración remota
  • Puerto 80 (HTTP) - necesario para Let's Encrypt y redirección
  • Puerto 443 (HTTPS) - acceso web de usuarios
Importante: Los puertos 80 y 443 deben estar abiertos a todo Internet (sin restricción de IP de origen) para que el certificado SSL de Let's Encrypt pueda validarse correctamente.

11.3 Conectividad entre servidores

  • El equipo de sincronización debe poder conectar al puerto 1433 del servidor ERP (origen)
  • El equipo de sincronización debe poder conectar al puerto 1433 del servidor Linux (destino)
  • Si las imágenes están en carpeta de red, el equipo de sincronización debe tener la unidad mapeada

12. Dominio y Certificado SSL

12.1 Registro DNS

Se necesita un registro DNS tipo A que apunte el subdominio elegido a la IP pública del router de la empresa:

stocks.empresa.com A <IP_pública>
Importante: Debe haber un único registro A para el dominio. Si hay múltiples registros A apuntando a diferentes IPs, el certificado SSL no se podrá generar correctamente.

12.2 Certificado SSL

Se utiliza Let's Encrypt (gratuito) para generar el certificado HTTPS. El certificado se renueva automáticamente cada 90 días mediante certbot.
