1. Introduccion y Alcance del Proyecto

RocaNET es un portal web que permite a los clientes de la empresa consultar en tiempo real el stock disponible de productos ceramicos, visualizar imagenes, descargar fichas tecnicas, y realizar solicitudes de material.

El sistema se alimenta de los datos del ERP existente de la empresa, sincronizando periodicamente las tablas de articulos, stocks, clientes, pedidos, albaranes y facturas hacia una base de datos replicada en un servidor dedicado.

Funcionalidades principales

  • Consulta de stocks en tiempo real con filtros avanzados (formato, serie, calidad, tono, calibre)
  • Visualizacion de imagenes de productos
  • Descarga de fichas tecnicas en PDF
  • Sistema de propuestas / solicitudes de material con generacion de PDF y envio por email
  • Consulta de pedidos, albaranes y facturas
  • Dashboard de estadisticas para administradores
  • Gestion de usuarios con roles (usuario, administrador, superusuario)
  • Soporte multi-idioma (espanol, ingles, frances)
  • Personalizacion visual por empresa (logo, favicon, tema de colores)
  • Acceso via API REST para integraciones externas

2. Arquitectura del Sistema

El sistema puede desplegarse en dos modalidades segun las necesidades y la infraestructura del cliente:

2.1 Modalidad A: Conexion directa al ERP

La aplicacion web se conecta directamente a la base de datos del ERP. No requiere sincronizacion ni servidor de base de datos adicional. Es la opcion mas sencilla cuando el servidor ERP es accesible desde el servidor web.

ElementoDescripcion
Servidor LinuxAplicacion web (Flask + Nginx + Redis) en contenedores Docker
SQL Server ERPLa aplicacion lee directamente de las tablas del ERP
VentajasSin replicacion, datos siempre en tiempo real, menos infraestructura
RequisitosEl servidor Linux debe tener conectividad al puerto 1433 del SQL Server ERP

2.2 Modalidad B: Base de datos espejo (replicada)

Se despliega un SQL Server adicional en el servidor Linux (Docker) y se replican periodicamente las tablas del ERP. La aplicacion web lee de la copia local. Es la opcion recomendada cuando:

  • El servidor ERP no es accesible desde Internet o desde el servidor web
  • Se desea aislar la carga de la web del servidor ERP de produccion
  • Se necesita mayor rendimiento en las consultas web
ElementoDescripcion
Servidor ERP (origen)SQL Server con las tablas de gestion (articulos, stocks, pedidos, facturas). Imagenes en carpeta de red o tabla de BD
Servidor Linux (destino)SQL Server 2025 en Docker con la copia de las tablas. Aplicacion web (Flask + Nginx + Redis)
SincronizacionScript BCP ejecutado desde un PC Windows con acceso a ambos servidores (tarea programada cada 30 min - 4 horas)
VentajasAislamiento total del ERP, funciona aunque el ERP este apagado, mejor rendimiento

4. Datos Necesarios del ERP

El sistema necesita acceso de solo lectura a las siguientes areas del ERP:

4.1 Datos maestros

  • Empresas: Codigo y nombre de las empresas gestionadas
  • Articulos: Catalogo completo de productos (codigo, descripcion, formato, modelo, unidad, estado web)
  • Formatos: Dimensiones y tipos de producto
  • Calidades: Niveles de calidad
  • Modelos: Series/colecciones de producto
  • Colores: Catalogo de colores
  • Tonos y Calibres: Variaciones de tono y calibre
  • Unidades de medida: M2, ML, unidades, etc.
  • Pallets: Tipos de pallet y configuraciones
  • Clientes: Datos de clientes (razon social, CIF, email, direccion)
  • Paises y Provincias: Tablas geograficas

4.2 Datos de stock y almacen

  • Stock por ubicacion: Existencias por articulo, calidad, tono, calibre, pallet
  • Configuracion de empaquetado: Piezas por caja, cajas por pallet, pesos
  • Codigos EAN13: Codigos de barras por articulo y calidad
  • Ubicaciones de almacen: Mapa de ubicaciones

4.3 Datos comerciales

  • Pedidos: Cabecera y lineas de pedidos de venta
  • Albaranes: Cabecera y lineas de albaranes de entrega
  • Facturas: Cabecera y lineas de facturas

4.4 Imagenes y documentos

  • Imagenes de articulos: Fotografias de producto (JPG). Pueden estar en tabla de base de datos o en carpeta de red
  • Fichas tecnicas: Documentos PDF con especificaciones tecnicas (opcional)

6. Imagenes de Articulos

Las imagenes de los articulos pueden provenir de dos fuentes segun la instalacion del ERP:

6.1 Opcion A: Tabla en base de datos

Las imagenes se almacenan como campo binario (IMAGE/VARBINARY) en una tabla del ERP. Se sincronizan junto con el resto de tablas mediante BCP.

6.2 Opcion B: Carpeta de red

Las imagenes se almacenan como ficheros JPG en una unidad de red compartida. El formato del nombre de fichero es:

CODIGO_ARTICULO@NUMERO.jpg Ejemplos: 0000000000004@1.jpg (primera imagen del articulo 0000000000004) 0000000000004@2.jpg (segunda imagen del mismo articulo) 0000000000012@1.jpg (primera imagen del articulo 0000000000012)

El script de sincronizacion:

  • Lee solo la imagen principal (@1) de cada articulo marcado como visible en web
  • Redimensiona las imagenes a un maximo de 800px de ancho para optimizar el almacenamiento
  • Solo inserta imagenes nuevas (no borra ni reemplaza las existentes)
  • Utiliza PowerShell para la lectura de ficheros y la insercion en base de datos
Requisito: El equipo que ejecuta la sincronizacion debe tener acceso a la unidad de red donde se almacenan las imagenes (por ejemplo Z:\).

8. Proceso de Instalacion

8.1 Preparacion del servidor Linux

  1. Instalar Ubuntu Server 22.04/24.04 LTS
  2. Configurar IP fija, DNS y zona horaria (Europe/Madrid)
  3. Instalar Docker y Docker Compose
  4. Crear usuario de aplicacion
  5. Abrir puertos necesarios en firewall (22, 80, 443, 1433)

8.2 Despliegue de SQL Server (Docker)

docker run -d \ --name sqlserver2025 \ --restart unless-stopped \ --network jobers-net \ -e 'ACCEPT_EULA=Y' \ -e 'MSSQL_SA_PASSWORD=<contrasena_segura>' \ -e 'MSSQL_PID=developer' \ -e 'TZ=Europe/Madrid' \ -p 1433:1433 \ -v sqldata:/var/opt/mssql \ mcr.microsoft.com/mssql/server:2025-latest
Zona horaria: Es imprescindible configurar TZ=Europe/Madrid tanto en el contenedor de SQL Server como en el de la aplicacion. Sin esta configuracion, las fechas se almacenaran en UTC.

8.3 Creacion de bases de datos

  1. Base de datos ERP replicada (ej: RocaNET_ERP): Contendra las tablas sincronizadas del ERP
  2. Base de datos de aplicacion (ej: ApiRestStocks): Contendra usuarios, parametros, configuracion, y las vistas SQL
  3. Base de datos central (ej: jobers_control_usuarios): Gestion centralizada de conexiones multi-empresa

8.4 Despliegue de la aplicacion (Docker Compose)

La aplicacion se despliega con Docker Compose incluyendo:

  • Contenedor backend (Python/Flask + Gunicorn)
  • Contenedor Redis (cache y sesiones)
  • Volumen para el frontend (servido por Flask)

8.5 Configuracion de Nginx (proxy inverso + SSL)

  • Redireccion HTTP a HTTPS
  • Certificado SSL via Let's Encrypt (certbot)
  • Proxy pass al backend en puerto 5000

9. Sincronizacion de Datos

La sincronizacion se ejecuta desde un equipo Windows de la red local mediante un script .bat programado como tarea de Windows.

9.1 Proceso de sincronizacion

  1. Verificacion de conexiones: Comprueba conectividad con servidor origen y destino
  2. Comparacion de esquemas: Si el numero de columnas de una tabla ha cambiado en el origen, recrea la tabla en el destino automaticamente (DROP + CREATE)
  3. Comparacion de datos: Calcula CHECKSUM_AGG de cada tabla en origen y destino. Si difiere, ejecuta TRUNCATE + BCP para actualizar
  4. Imagenes: Compara ficheros de la carpeta de red con registros existentes. Solo inserta las imagenes nuevas
  5. Finalizacion: Compacta el log de transacciones y registra la fecha de ultima sincronizacion

9.2 Frecuencia recomendada

FrecuenciaUso recomendado
Cada 30 minutosEntornos con actualizacion frecuente de stock
Cada 2-4 horasEntornos con volumen moderado
1 vez al dia (noche)Entornos con poco cambio o tablas muy grandes
Nota: Las tablas se leen con WITH (NOLOCK) para minimizar el impacto en el rendimiento del ERP origen. No obstante, la sincronizacion de tablas muy grandes (>50.000 registros) puede generar carga momentanea en el servidor ERP.

9.3 Compatibilidad BCP

El script utiliza BCP en formato -w (wide character) que garantiza compatibilidad entre diferentes versiones de SQL Server (2008, 2012, 2019, 2025) y entre plataformas Windows y Linux.

10.2 NAT / Port Forwarding (router)

Para acceso desde Internet, el router de la empresa debe redirigir los siguientes puertos a la IP interna del servidor Linux:

  • Puerto 22 (SSH) - opcional, solo para administracion remota
  • Puerto 80 (HTTP) - necesario para Let's Encrypt y redireccion
  • Puerto 443 (HTTPS) - acceso web de usuarios
Importante: Los puertos 80 y 443 deben estar abiertos a todo Internet (sin restriccion de IP de origen) para que el certificado SSL de Let's Encrypt pueda validarse correctamente.

10.3 Conectividad entre servidores

  • El equipo de sincronizacion debe poder conectar al puerto 1433 del servidor ERP (origen)
  • El equipo de sincronizacion debe poder conectar al puerto 1433 del servidor Linux (destino)
  • Si las imagenes estan en carpeta de red, el equipo de sincronizacion debe tener la unidad mapeada

11. Dominio y Certificado SSL

11.1 Registro DNS

Se necesita un registro DNS tipo A que apunte el subdominio elegido a la IP publica del router de la empresa:

stocks.empresa.com A <IP_publica>
Importante: Debe haber un unico registro A para el dominio. Si hay multiples registros A apuntando a diferentes IPs, el certificado SSL no se podra generar correctamente.

11.2 Certificado SSL

Se utiliza Let's Encrypt (gratuito) para generar el certificado HTTPS. El certificado se renueva automaticamente cada 90 dias mediante certbot.
